# 测试框架与流程方案（jimo 低代码平台）

> 结合「芯片验证方法借鉴到软件测试」的思想，与近期 autocode 网格页（grid）开发中踩到的真实 bug，
> 给出本项目量身定的测试分层方案。目标：把「手动生成一张表、刷新看效果」的验证，
> 升级为**自检（self-checking）**的自动化回归。
> 日期：2026-06-29 · 版本：v3（L0/L1/L2/L3 均已落地，bug-5 真实根因已修复）

---

## 一、背景与核心洞察

### 1.1 本项目的「DUT」是分层的

| 层级 | 代码 | 风险特征 |
|---|---|---|
| 平台代码 | 生成器、鉴权(JWT/Casbin)、BPM 集成、软删除 | 普通业务风险 |
| **生成产物** | autocode 按 schema 生成的 NestJS 模块 + React 页面 | **一处 bug → 污染所有生成的模块** |

**生成器是临界点**：它改一次，所有已生成和将生成的模块都受影响。这是本项目区别于普通 CURD 系统的最大风险面，也是测试投入 ROI 最高的地方。

### 1.2 借鉴报告的第一性原理

> 测试平台必须能**自己判断对错（self-checking）**，人只负责设计场景和分析漏洞。

把芯片验证里「参考模型（Reference Model）+ 计分板（Scoreboard）」两个角色从脑子搬到代码里。
我们的 grid 开发过程中，**手动 tsc 编译生成页 + 手动对 `.umirc.ts` 跑正则验证**，本质上就是人脑在当 Reference Model + Scoreboard——这套流程必须自动化。

## 二、Bug → 测试层映射表（整个框架的依据）

近期 grid 开发修过的 bug，反推到「该由哪层测试拦住」：

| 这次修的 bug | 拦住它的层 | 具体断言 / 不变量 |
|---|---|---|
| detail 路由缺 `./` + regenerate 累积重复路由 | **L0** | `stripTableRouteBlocks` 纯函数测试：任意路由集合下只删目标表、绝不误伤他表；重复块全清 |
| 删除残留 detail.tsx / map.tsx | **L0** | 源码回归守卫：`history.service.ts` 的删除清单含 `detail.tsx`、`map.tsx` |
| `pageSize: 9999` > 100 | **L0 + L2** | L0：生成产物不含 `pageSize`>100；L2：请求 `pageSize=9999` → 400 |
| 日期 Invalid Date（dayjs） | **L0** | 有 timestamp 字段 ⟺ 产物含 `import dayjs` + `TIMESTAMP_FIELDS` + 转换 |
| grid 缺 agent 按钮 | **L0** | `agentConfig.enabled ⟺` 产物含 `EntityAgentPanel` + `RobotOutlined` + `businessType=` |
| 新增行刷新后丢失（POST 发了但 500：unique 列空串撞唯一索引） | **L0 + L3** | L0：产物 `onValuesChange` 对新行调用 `scheduleCreate`；**且 unique 字段产物必 `.notNull()` 且无 `.default('')`**（否则空串撞部分唯一索引→POST 500）；L3：E2E 行为验证 |

**5 个里 4 个能被 L0（又快又确定的产物断言）单独拦住**。这就是分层依据，也是 v1 先做 L0 的原因。

## 三、四层测试框架（映射 UVM 概念，不背重壳）

### L0 — 生成器产物契约测试 = Reference Model + Scoreboard　🟢 v1 落地

本项目的「自检测试平台」，ROI 最高。

- **输入矩阵**：用一组代表性 `AutoCodeDto`（pageType × 字段类型组合 × agent/approval 开关）喂给生成器。
  v2 升级为 `fast-check` 受约束随机生成（= 报告里的 Constrained Random），自动覆盖「grid + point」「document + timestamp」等人工想不到的组合。
- **计分板（自动判对错）**：
  1. **契约断言**：产物源码字符串必须满足一组不变量——上表每条 bug 都变成一条 `expect(src).toMatch(...)` / `not.toMatch(...)`。
  2. **纯函数单元测试**：把生成器里 bug 易发的纯逻辑（路由剥离 `stripTableRouteBlocks`、命名 `deriveNames`）抽成纯函数直接测。
  3. **源码回归守卫**：对关键的「一次性字符串修复」（detail 路由 `./` 前缀、删除清单）做源码文本断言，防止被无意改回。
- 速度：纯字符串 + 正则，**秒级**。改生成器时强制全量回归。

> 关于「真实 tsc 编译生成页」：生成页 import `@/services/*`、`@ant-design/pro-components` 等 web 侧依赖，
> 无法在 server 的 Jest 环境直接编译。L0 用**契约断言**替代（断言 import 是否齐全、模式是否正确），
> 能抓到同类问题（如 relation 选项函数漏 import）。完整的 web 编译校验留作 L0+（需独立 fixture 工程）。

### L1 — 单元 + 属性测试 = 不变量 / 蜕变关系　🟡 已有基础

- 已有：`formula-evaluator.spec.ts`（31 个 Jest 用例）、`er-graph.util.spec.ts`。保留扩展。
- 目标不变量（v2 用 `fast-check`）：
  - 软删除：所有查询带 `isNull(deletedAt)`。
  - 命名往返：`deriveNames` 一致性。
  - PATCH 单字段不污染关联（单元格自动保存的核心依赖）。

### L2 — API 契约测试 = 业务规则的 Reference Model　🟡 规划中

报告说「中层 API 测试性价比最高」。NestJS 用 `supertest` + 专用测试库（testcontainers PG）：
- 生成模块 CRUD 往返。
- **PATCH 单字段前后比对关联数据不变**（Scoreboard）。
- 鉴权参考模型：非 owner 不能改、Casbin 策略、token 黑名单——用独立纯函数重算期望值再比对。
- `pageSize` 越界校验（>100 → 400）。

### L3 — E2E 冒烟 = Driver（Page Object），只测 P0　🟡 规划中

- Playwright，**只覆盖 P0 主流程**：
  - 登录 → 生成模块 → 打开 list/grid/document → **新增行→填必填→刷新→行还在**（bug 5）→ 单元格编辑持久。
- Page Object 封装每种 pageType（= Driver），UI 改版只改 Driver，用例不动。
- 规模控制，夜间 / 发版跑。

## 四、流程与 CI（变更影响分析）

| 时机 | 跑什么 | 触发条件 |
|---|---|---|
| pre-commit | L0 + L1 | 秒~分钟级 |
| PR CI | + L2 | 合并前 |
| nightly / 发版 | + L3 全量 | 定时 |
| **改了生成器** | **L0 全矩阵强制回归** | `autocode-*-generators.ts` / `entrypoint.service.ts` 变更 |

最后一条最关键：生成器是「改一次影响全部」的临界点，回归必须自动化、矩阵化。

## 六、L0 v1 详细设计（本次落地）

### 6.1 为可测性做的最小重构

把路由剥离逻辑从 `EntrypointService` 的私有方法抽成**纯函数**：
`stripTableRouteBlocks(content: string, pageDir: string): string`（新建 `route-lifecycle.ts`）。
`entrypoint.service.ts` 改为调用该纯函数。纯函数无 DI 依赖，可直接单测。

### 6.2 测试文件

| 文件 | 内容 |
|---|---|
| `src/modules/autocode/autocode-frontend-generators.spec.ts` | 矩阵 DTO → 生成 grid/list/document 页 + service，对每个产物跑契约断言（pageSize、dayjs、relation import、agent↔enabled、businessType、新行 scheduleCreate） |
| `src/modules/autocode/route-lifecycle.spec.ts` | `stripTableRouteBlocks` 纯函数：删目标全清（含重复）、不误伤他表、前缀安全（`lc/order` vs `lc/order-item`）；+ 源码回归守卫（`./` 前缀、删除清单） |

### 6.3 契约断言清单（Scoreboard 规则）

每条对应一个已修 bug，防止回归：

- grid 产物含 `const PAGE_SIZE = 100`，不含 `pageSize: 9999`。
- grid 有 timestamp 字段 ⟺ 含 `import dayjs` + `TIMESTAMP_FIELDS` + `dayjs(r[k])`。
- grid 有 m2o relation ⟺ 含对应 `get<Table>Options` import。
- grid `agentConfig.enabled` ⟺ 含 `EntityAgentPanel` + `RobotOutlined` + `AI 助手` + `businessType="<tableName>"`。
- grid `onValuesChange` 对新行调用 `scheduleCreate`（新行自动创建）。
- **unique 字段产物必 `.notNull()` 且无 `.default('')`、create DTO 含 `@IsNotEmpty`**：unique 列若可选会落空串 default，多条空串撞部分唯一索引 → POST 500 → 行丢失（bug-5 真实根因，非"没发 POST"；由 `enforceUniqueRequired` 在生成器层强制 unique→required）。
- `stripTableRouteBlocks`：剥离后他表路由数不变、本表路由清零、重复块全清、前缀不误伤。
- 源码守卫：`entrypoint.service.ts` 含 `'./${n.pageDir}/detail'`；`history.service.ts` 删除清单含 `detail.tsx`、`map.tsx`。

## 七、必须泼的冷水（成本/收益）

借的是**自检测试平台 + 受约束随机 + 覆盖率驱动**三个思想，**不**照搬 UVM 重型架构：
- 不追求 100% 覆盖率，投入产出比会害死人。
- 不为「能改的代码」搭多层 Agent/Sequencer（那是给「改不了的芯片」的奢侈品）。
- UI 只做 P0 冒烟，不做全面 UI 自动化。
- LLM-as-judge 只用于软判断（约定、可读性），数值/状态/契约永远优先确定性比对。

## 八、路线图

- **v1 ✅ 已落地**：L0——生成器产物契约测试 + 路由生命周期纯函数测试。拦住近期 5 个 bug 中的 4 个。
- **v2 ✅ 已落地**：
  - 受约束随机：`fast-check` 把生成器矩阵从手写升级为属性测试（200 次/属性），并**立刻发现** dayjs 不变量漏了 `editable` 维度。
  - L1 不变量属性测试：`deriveNames` 命名一致性、`stripTableRouteBlocks`「只删目标表」。
  - L2（DB-free）：`PaginationDto` 校验契约（pageSize 上限——那个 bug 的契约层守卫）。
  - L2（DB-backed，共享测试库 `lowcode_test`）：`StudentService` 真服务 + 真 Postgres，覆盖 CRUD 往返、**部分更新不污染其它字段（单元格自动保存的核心不变量）**、软删除隔离、唯一约束。门控于 `RUN_L2_DB=1`，`pnpm test` 不触库。
- **v3（本次）**：
  - L2 HTTP 全链路 ✅：supertest 起 Nest Testing app（StudentController + ValidationPipe + 前缀），覆盖 `pageSize>100 → 400`、create/list、部分 PATCH。**并发现一个真实缺口**：`main.ts` 的 `ValidationPipe(skipMissingProperties:true)` 会让缺失必填字段的请求绕过 `@IsNotEmpty`，落到 DB 层报 500（非干净的 400）——已在测试中以注释固化，建议后续把 `skipMissingProperties` 改 false。
  - O2M 关联不变量 ✅：生成器产物契约——断言生成的 `update()` 用 `if (dto.<o2m> !== undefined)` 守卫关联写入（单字段 PATCH 不会清空子表/替换 M2M）。
  - L3 Playwright 冒烟 ✅ 已写好并跑通（`apps/web/playwright.config.ts` + Page Object + P0 spec：登录、grid 挂载、**新行刷新后留存** bug-5 回归）：`pnpm test:e2e` → 2/2 passed（`@playwright/test` 已装）。运行前提：dev 栈已起（前端 :8000 + 后端 :8888）。
  - **bug-5 真实根因修复（unique→required）** ✅：本轮用 Playwright MCP + docker psql 定位——bug-5 并非"没发 POST"，而是 POST 返回 500（unique 列 `default('')` 空串撞部分唯一索引 `idx_*_active`）。生成器层加 `enforceUniqueRequired`（`generateSchema`+`activeFields`+AI propose 兜底），L0 补 4 条断言（unique 字段 `.notNull()`/无 `.default('')`/`@IsNotEmpty`）。**修正了本节原先对 bug-5 的错误归因**，以及原 L3 注释"装不上、未能运行"的过时描述。

## 一句话总结

把芯片验证的「**自检 + 随机 + 覆盖率**」三魂，落到本项目的最高风险面——**代码生成器**——上：
先用 Jest 字符串契约断言把生成产物变成 self-checking（L0），再逐层加随机、API、E2E，
而不必背上 UVM 的重型躯壳。

## 九、现状校正（2026-07-11）

本方案自称 v1~v3"均已落地"，实际运行中暴露出多处与文档不符的 gap。以下逐层校正，如实记录。

### L0 — 真有 152 条，但一度有 3 条红断言

- L0 现有 **10 个 spec 文件、152 条测试全绿**(`pnpm test` 实测)。
- 2026-07-11 前存在 **2 个测试 3 行红断言**(`autocode-backend-generators.spec.ts`)，原因是 `lc/` 命名空间重构将生成器产出的文件名改为 `lc-` 前缀(`lc-student.service`)，而这 3 个断言仍期望旧的无 `lc-` 命名。**已修**(改为 `./lc-student.controller` / `./lc-student.service`)，现在全绿。
- **教训**：生成器命名是 L0 契约测试的敏感轴——改了命名必须同步更新断言，否则 L0 红测试被晾着等于零保护。

### L2（DB-backed）— 能跑，但需本地配置

- L2(`lc-student.service.contract` / `lc-student.http.contract`,gated by `RUN_L2_DB=1`)本质上是**按 entity 生成的测试**：模板在 `autocode-test-generators.ts`，生成产物在 `modules/lc/<pkg>/<entity>/`。
- 初始版本硬编码 `localhost:5432` 和 `lowcode_test` 库名。**已修**(2026-07-11)：URL 改为读取 `TEST_DATABASE_URL` / `DATABASE_URL` 环境变量（有默认 fallback）。新增 `pnpm run db:test:setup` 便捷建库脚本。
- **运行前提**：① `DATABASE_URL` 指向 Postgres、② `RUN_L2_DB=1`、③ 被测 entity 的 Drizzle schema 已注册在 `db/schema/index.ts`（`lc/` schema 是 gitignored 的生成产物，注册由其 `EntrypointService.updateSchemaIndex` 自动完成）。
- **现状**：2026-07-11 用 `lc-student` demo 验证通过(service contract 4/4 + http contract 4/4,需临时注册 schema export)。L2 不进 CI(需 Postgres 服务容器)。

### L3（Playwright）— 配置存在，需要运行栈

- `apps/web/playwright.config.ts` + `e2e/students-grid.smoke.spec.ts` 存在，`@playwright/test` 已装。
- 运行前提：dev 栈在线（前端 :8000 + 后端 :8888）+ `playwright install chromium`。不进 CI，适合本地验证 / 夜间。

### CI — 此前不存在，已补

- **此前完全没有 CI**(无 `.github/workflows/`)。L0+L1 的回归只在有人记得手动跑时才起作用。
- **已补**(2026-07-11)：`push` 到 `main` + PR 时，GitHub Actions 自动跑 `pnpm test`(L0+L1,不触库)。工作流见 `.github/workflows/test.yml`。
